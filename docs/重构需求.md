# adminApp 业务逻辑与 UI 组件拆分重构需求文档

**简介**：将当前单一站点（Kellogg）重构为可切换站点的多站点架构。

**问题**：1. 当前的管理后台仍然是单一站点，诸多已经完善的业务逻辑无法直接复用，2. adminApp中的组件将业务逻辑与UI样式完全堆砌在了一起，导致维护困难，

**目标**：1. 将底层通用业务逻辑与专属 UI 适配层分离，提升代码可读性与业务逻辑复用率。2. 将业务逻辑代码从UI组件中拆分，方便后续修改UI，改变样式风格。

**文件目录**：底层通用的业务逻辑集中在src/core中，如context内容管理、数据请求/上传更新业务、通用数据类型等等。而站点专属资源则放在`site-package/{site-name}`中。

**实现效果**：只需要在组件'src\site-package\index.tsx'中，将路径切换至对应的站点，即可完成站点的切换。

- 典型的core组件 `src\context\ContentContext.tsx` 由于它没有触及站点专属的 blokcs积木块UI和后台UI组件，但是要注意的是 kellogg是站点专属的名称，所以要处理一下。
- 典型的站点专属组件 `src\admin\pageBuilder\propsEditors` 每个editor都对应一个 block积木块组件，

## 数据类型说明：
**通用的数据类型**，如：Product、Blog、CompanyInfo等等，这些都可以作为通用的数据类型。
**专属的资源（数据）**，实际上专属的数据类型就是blocks积木块的数据类型、积木块样式UI、后台样式UI、styles风格等等。

##  验收标准
1. core中的业务逻辑不得依赖站点资源，站点组件可以调用core中的业务逻辑，的专属的业务逻辑，如：数据类型解析/聚合，UI。
2. 业务和UI必须拆分，且逻辑清晰。
3. 重构后文章加载、编辑、保存草稿、图片粘贴上传及发布功能表现与之前完全一致。
4. 重构后的代码在 TypeScript 类型校验下无任何 Error/Warning 报错。

## 重要：
**UI组件（Eidtor）**：由于后台管理界面的UI不需要像 blocks积木块纯UI组件那样在多个项目之间进行复用，所以可以容忍一些项目/站点的依赖。
