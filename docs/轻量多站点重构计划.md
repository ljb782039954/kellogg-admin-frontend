# adminApp 轻量多站点重构计划

本文档基于 [重构需求.md](./重构需求.md) 编写，目标是在不破坏当前可运行状态的前提下，将单一 Kellogg 后台逐步整理为“通用业务核心 + 站点包适配层”的结构。

## 核心判断

当前代码已经可以正常运行，因此本次重构不追求一次性大规模搬迁，也不追求过度抽象。更稳妥的方向是：

- 先建立边界，再小批量迁移。
- 优先保留现有行为和路由。
- 通用业务逻辑进入 `src/core`。
- Kellogg 专属 UI、blocks、props editor、组件注册表、样式资源进入 `src/site-package/kellogg`。
- `src/site-package/index.tsx` 作为当前站点入口，只负责选择并导出当前启用的站点包。

一句话概括：业务流程复用，站点 UI 留在站点包里。

## 目标目录边界

### `src/core`

放置不依赖具体站点资源的通用逻辑。

适合迁入：

- 内容管理上下文：例如 `ContentContext`
- 语言上下文：例如 `LanguageContext`
- API 请求封装：例如 `api.ts`
- 图片上传、查重、链接处理等通用工具
- 通用数据类型：例如 `Product`、`Blog`、`CompanyInfo`、`Review`
- 与发布、草稿保存、媒体上传等流程相关的业务逻辑

限制：

- 不 import `src/site-package/**`
- 不 import Kellogg 专属 blocks UI
- 不 import Kellogg 专属 props editor
- 不持有具体站点的组件注册表实现

### `src/site-package/kellogg`

放置 Kellogg 站点专属内容。

适合迁入：

- blocks 积木块 UI
- blocks 对应的数据类型或解析逻辑
- `propsEditors`
- `componentRegistry`
- `blocksContentPreview`
- 站点默认配置、样式风格、站点资源
- 可以带有 Kellogg 依赖的后台 editor UI

说明：

后台管理界面的 Editor UI 不需要像前台 blocks 一样跨项目复用，因此可以容忍站点依赖。这里不强行把 Editor UI 拆成完全无站点依赖的抽象组件。

### `src/site-package/index.tsx`

作为站点切换入口。

目标效果：

```ts
export * from './kellogg';
```

后续新增站点时，只需要将这里切换到另一个站点包即可。

## 建议迁移顺序

### P0：建立站点包契约，不移动业务代码

目标：先把“站点包应该提供什么”定义清楚。

建议动作：

1. 在 `src/site-package/kellogg` 中创建最小入口文件。
2. 从站点包导出当前站点名称、组件注册表、页面构建相关资源。
3. 让 `src/site-package/index.tsx` 只转发当前站点包。

验收：

- 不改变现有页面行为。
- TypeScript 通过。
- 不改路由，不改数据保存逻辑。

### P1：迁移站点专属配置

目标：先迁移最容易识别的站点资源。

建议迁移：

- `src/config/componentRegistry.ts`
- `src/config/blocksContentPreview/**`
- 与 blocks 默认内容、展示样式强相关的配置

保守策略：

- 旧路径可以暂时保留 re-export，避免一次性修改大量 import。
- 新代码优先从 `src/site-package` 入口读取。

验收：

- 页面编辑器仍能正常显示组件列表。
- 单例 block 规则仍然有效。
- 只运行页面编辑器/组件注册表相关测试，必要时再运行 TypeScript。

### P2：迁移 blocks 与 propsEditors

目标：将真正的站点专属 UI 收敛到 Kellogg 站点包中。

建议迁移：

- `src/components/blocks/**`
- `src/admin/pageBuilder/propsEditors/**`
- 与 block 类型强绑定的预览、缩略图、属性编辑分发逻辑

保守策略：

- 先迁移目录，再用 re-export 兼容旧 import。
- 一次只迁移一组关联文件，例如先迁 `blocks`，再迁 `propsEditors`。
- 不改变 editor 的表单行为和字段结构。

验收：

- 页面加载、block 添加、删除、排序、编辑行为一致。
- 与 page builder 相关的测试通过。
- TypeScript 无 Error/Warning。

### P3：迁移 core 通用业务

目标：把可复用的业务流程移动到 `src/core`，但不改业务语义。

建议迁移：

- `src/context/ContentContext.tsx`
- `src/context/LanguageContext.tsx`
- `src/lib/api.ts`
- `src/lib/image.ts`
- `src/lib/linkUtils.ts`
- 通用 types

注意：

- `ContentContext` 可以依赖通用类型和 API，但不应该直接依赖具体站点 blocks UI。
- 如果 `ContentContext` 当前使用了 `kellogg` 之类站点名称，需要改成从站点包或配置注入。
- 不要为了“纯净”拆散已经稳定的业务流程。

验收：

- 文章加载、编辑、保存草稿、图片粘贴上传、发布功能表现一致。
- 只运行与 context、blog、media、publish 相关的测试。
- TypeScript 无 Error/Warning。

### P4：整理 App 入口与路由适配

目标：让 App 使用 core provider 和站点包导出的管理界面资源。

建议动作：

1. `App.tsx` 保持现有路由结构。
2. 将站点专属页面组件逐步从站点包导入。
3. 对确实通用的 Shell、Provider、基础布局再考虑放入 `core`。

保守策略：

- 不重写路由系统。
- 不引入复杂的插件式路由注册，除非第二个站点真的需要。
- 当前阶段允许 Kellogg 的后台页面仍然按现有路由显式挂载。

验收：

- 所有原有后台路由仍可访问。
- 发布按钮、侧边栏、媒体库、文章管理行为一致。
- TypeScript 通过。

## 不做的事情

为了避免再次走向复杂化，当前阶段明确不做：

- 不设计复杂插件系统。
- 不把所有 Editor UI 强行抽象为跨站点组件。
- 不一次性迁移整个 `src/admin`。
- 不重写 ContentContext 的数据流。
- 不改变 API 协议。
- 不改变 KV/D1 数据结构。
- 不改变现有路由语义。
- 不进行生产构建验证。
- 不运行全量测试。

## 推荐验证方式

每个小批次只做相关验证：

- 配置或类型迁移：运行 TypeScript。
- page builder 迁移：运行 page builder / component registry 相关测试。
- blog 或 media 相关迁移：运行 blog / media 相关测试。
- context 或 API 迁移：运行 context/API 相关测试。

最终验收再统一确认：

- TypeScript 类型检查无 Error/Warning。
- 文章加载、编辑、保存草稿正常。
- 图片粘贴上传正常。
- 发布触发正常。

## 第一批建议执行内容

第一批只做低风险边界落地：

1. 完善 `src/site-package/kellogg` 入口。
2. 让 `src/site-package/index.tsx` 指向 Kellogg。
3. 迁移或 re-export `componentRegistry`。
4. 保持旧路径兼容，暂不大改 import。
5. 运行 TypeScript 校验。

这样做的收益是：项目马上拥有明确的站点切换入口，但不会立刻触碰复杂的数据流和编辑器行为。
